Do arquivo imperfeito à auditoria publicada — com segurança, pagando pouco

SaaS para escritórios de contabilidade: upload de planilhas, mapeamento, regras determinísticas, revisão em equipe, publicação e compartilhamento com senha. Este documento é a visão navegável; a especificação integral (SQL, rotas, telas) está em docs/PRD.md.

Decisões estruturais da v2

O que mudou em relação ao PRD v1 — cada uma verificada contra docs oficiais (Context7, jul/2026).

Backend

Supabase fecha a questão

Postgres + Auth + Storage + Edge Functions + Realtime. RLS multi-tenant por escritório. TypeScript de ponta a ponta — sem segundo deploy.

Processamento

Parse no browser, regras no banco

Edge Functions têm 2s de CPU/request (verificado). O Web Worker parseia; as 7 regras vivem versionadas em SQL — cada resultado grava fórmula, valores e versão.

Segurança

Share com snapshot imutável

Link + senha bcrypt + rate limit em tabela + erro genérico. O cliente só enxerga o snapshot publicado — nunca dados vivos.

Cobrança

Stripe pronto, piloto grátis

Checkout + Portal + webhook idempotente + gate de inadimplência. Trial de 90 dias sem cartão via seed; ligar preço é decisão de negócio.

Equipe

Colaboração na v1

4 perfis (proprietária, contador, analista, cliente externo), convites por e-mail, matriz de permissões — RF-001–068 do anexo de arquitetura.

UX

Auditoria como workspace

Cabeçalho persistente + abas; tabela de inconsistências com toolbar como centro; revisão em painel lateral; dashboard por exceção (fila antes de gráfico).

Perfis

Permissões simples, adequadas ao trabalho em equipe. Matriz completa no anexo.

PerfilPodeNão pode
Proprietária (owner)Tudo: equipe, todas as auditorias, aprovar, publicar, revogar, logs—
Contador (accountant)Criar/editar auditorias, mapear, revisar, justificar, compartilhar quando autorizadoGerenciar equipe/configurações críticas
Analista (analyst)Clientes, rascunhos, upload, mapeamento, notas internasAprovar, publicar, gerar/revogar links
Cliente finalVer UMA auditoria publicada via link+senha; PDF quando permitidoCriar conta, editar, ver dados internos

Fluxo principal

Jornada por etapas dentro de um workspace — não um CRUD.

1. CriarCliente, período, responsável.
2. ArquivosXLSX/CSV, 20MB, 5 por auditoria (TUS).
3. MapearSugestão + confirmação; reuso por cliente.
4. ProcessarWorker normaliza; SQL executa as regras.
5. RevisarJustificar, ocultar, comentar — em equipe.
6. AprovarOwner; zero pendências.
7. PublicarSnapshot imutável versionado.
8. CompartilharLink + senha + expiração + acessos.
Princípio inegociável: nenhuma linha é descartada silenciosamente — toda linha tem status e mensagem, garantido por constraint de banco (msg_required em normalized_rows).

Backend — Supabase (§8 do PRD)

Schema completo, RLS, pipeline, share e Stripe — com SQL pronto no PRD.md.

Dados

Fluxo

  • Original → Storage privado (TUS, sha-256 registrado).
  • Worker → lotes JSON → ingest_rows (idempotente, re-validação server).
  • run_rules → 7 regras SQL versionadas → rule_results com fórmula+valores.
  • Progresso via Realtime (canal privado por auditoria).
  • Máquina de estados com transições legais tabeladas.
Segurança

Share e RLS

  • RLS em todas as tabelas; tenancy por escritorio_id no JWT (app_metadata).
  • Cliente externo: zero políticas anon — só RPCs redeem_share/get_shared_snapshot.
  • Senha bcrypt (pgcrypto), rate limit (5/15min por token+IP), erro genérico, ip_hash com pepper.
  • Snapshot imutável (trigger rejeita UPDATE/DELETE); log de acessos.
  • Backups Pro diários + runbook de restore em 5 passos.

Design (§10 do PRD)

14 superfícies especificadas com hierarquia, estados e microcopy. Tema do template intocado — única extensão: tokens de status.

Sistema

Regras que valem em toda tela

  • Hierarquia fixa: cabeçalho → próxima ação → ≤5 KPIs → toolbar → conteúdo.
  • Status = ícone + texto + cor (nunca só cor; formas distintas por severidade).
  • Parcial é estado de 1ª classe ("1.180 de 1.252 linhas" com motivo por linha).
  • Modal só para confirmação; fluxos são páginas.
  • WCAG 2.2: foco nunca sob a toolbar sticky, alvos ≥24px, senha colável no gate.
Duas vozes

Interno × cliente

Interno: "Conta 1.1.2.03 (jan/2026): saldo inicial + débitos − créditos difere do saldo final em R$ 470,00" — com fórmula, valores e origem Extrato.xlsx:214.

Cliente (/r/:token): "O saldo informado não corresponde aos movimentos registrados no período" — zero jargão, valores lado a lado, explicação do escritório.

Roadmap (§11)

Seis fases verticais; front e back andam juntos; cada fase termina verificável.

F1

Fundação

Migrations core + RLS, invite-user, seed do trial; auth Supabase no front, guard por perfil, clientes. Aceite: login/convite + RLS testada + CRUD no browser.

F2

Auditorias

register_file/save_mapping/storage; lista + workspace + máquina de estados. Aceite: criar → subir (TUS) → mapear; transição ilegal recusada.

F3

Processamento

ingest_rows + run_rules + regras v1; stepper + worker + progresso ao vivo. Aceite: arquivo real 20MB, 100% das linhas com status.

F4

Revisão

Tabela de inconsistências + Sheet + vistas + bulk; dashboard por exceção. Aceite: revisão completa por teclado; aprovação bloqueada com pendências.

F5

Publicação & Share

Snapshot + shares + /r/:token + PDF. Aceite: cliente acessa em aba anônima; 5 senhas erradas bloqueiam 15min.

F6

Piloto

Stripe + gate + CI + a11y + suíte completa. Aceite: checkout test mode ponta a ponta; restore ensaiado; critérios do anexo todos verdes.
